4.23 重拾Java之异常与多线程

  1. 使用throws关键字和throw的区别 throws->声明一个方法可以不用处理异常类,但是调用的是必须处理异常,主方法也可以用此关键字,交给JVM处理异常,主方法内内不用处理异常 格式:public return FuncName() throws exception throw->可以自己抛出一个手工产生的异常实例化对象new exception()
  2. 自定义异常类 在MyException类中声明构造方法,并且用super()调用父类的构造方法,再利用throw关键字抛出new MyException("ExceptionInfo")即可
  3. Java要将一个包提供给其他人使用,一般是将其压缩成一个jar包.使用的时候需要在classpath出注册这个jar SET CLASSPATH = .;F:\TEST\my.jar
  4. 多线程有两种实现方式:Thread类和Runnable接口
->继承Thread类
class MyThread extends Thread{
    private String name;
    public MyThread(String name){
        this.name = name;
    }
    public void run(){
        System.out.println("123");
    }
}
调用:
MyThread myTh1 =  new MyThread("线程A");
MyThread myTh2 =  new MyThread("线程B");
myTh1.start();//start()方法才是开启多线程
myTh2.start()

->实现runnable接口
class MyThread implements Runnable{
    private String name;
    public MyThread(String name){
        this.name = name;
    }
    public void run(){
        System.out.println("123");
    }
}
调用:
MyThread myTh1 =  new MyThread("线程A");
MyThread myTh2 =  new MyThread("线程B");
//Thread默认提供传入Runnable子类的构造方法,完成多线程开启
Thread A = new Thread(myTh1);
Thread B = new Thread(myTh2);
A.start();//start()方法才是开启多线程
B.start()

->它俩的关系是,Thread是Runnable接口的子类,通过继承Thread的时候必须重写run()方法;

->同时,使用Runnable是可以资源共享的,尽量使用Runnable接口;

->Java中所有线程都是同时启动的,谁先抢到CPU资源谁先运行;

->synchronized可以声明代码块,也可以用来声明同步方法;

4.22重拾java之开端

前一段时间的纠结可以告一段落了,我还是选择了重拾java这一条路,毕竟这曾经是让我入门编程的语言.
下了一篇电子书,快速得浏览了一下一些基础和面向对象的实例之后,之前有些疑惑也豁然开朗.同时心中也难免产生一些和OC的对比.举个栗子:

OC写一个构造方法
- (instance)initWithName:(NSString *)name{
    self = [super init];
    if (self){
        self.name = name;
    }
    return self;
}

Java写一个构造方法:
class TestDemo{
    public TestDemo (String name){
        this.name = name;
    }
}
-----------分隔线---------------
OC中写一个单例:
+(DBManager *)singleton; (.h文件中)
.m文件中的实现:
+(DBManager *)singleton{
　　Static DBManager *singleton = nil;
      static dispatch_once_t token;
      dispatch_once(&token,^{
              if(singleton == nil){
　　         singleton = [[DBManager alloc]init];
        }
   } );
  return singleton;
}

Java中写一个单例:
class Singleton{
    private static Singleton instance = new Singleton();
    private Singleton(){   
    }
    public static Singleton getInstance(){
        return instance;
    }
}

其实都是异曲同工.

->关于self和super都必须放在放在构造方法的第一行.

->final关键字修饰的类,方法,属性不能被继承,重写和修改,修饰的常量必须是'大写'.

->抽象类中,需要在类名前加一个abstract,声明抽象方法只需声明无须事先,必须在权限后加abstract关键字.并且抽象类不能用final关键字,必须被继承,抽象方法不能用private关键字,因为必须被子类重写.

->interface中定义的是全局常量和public抽象方法(不写public也默认为此,而非default)

->instanceof类似OC中的isKindOfClass,可判断实例类型

->模板设计:父类定义一些可能用到的方法作为模板,子类按照模板实现方法就行了.

->避免空指针小技巧,将str.equalsto("123")改成"123".equalsto(str)

->接口也可以被实例化,接受的是实现接口抽象方法的类,类似于父类和子类的关系

->java中的适配器模式:接口方法过多,那么声明一个抽象类实现接口所有接口方法,但是该类实现的方法都为空实现,最后声明一个类继承这个抽象类,实现需要的方法.

4.21 惠州小桂苑游览

本来是当晚就会记录这次游览旅程的,但是回来当晚停电了,所以这事就鸽了,现在补起来.
这是公司组织的一次团体活动,总体内容包括:自行车骑行,坐快艇登陆东升岛游览民居,团队野炊,自己生火做饭.
针对这次活动呢,我是非常期待的,毕竟作为一个程序员,长期面对电脑,出去看看海也还是不错的.当天天气十分阴沉,看样子随时会下雨,用波哥的话说就是:虫鸟低飞,肯定下雨.事实证明后来确实下了......
到达目的地之后的第一项自然就是自行车骑行,我和波哥选择了一辆双人骑的自行车,原因很简单,波哥不会骑自行车,嘿嘿.于是乎我就带着波哥去海边兜了一把风.大海是我比较向往的地方之一,这次我是真的体会到了什么叫一望无际的大海,那宽广,那包容,还有那咸咸湿湿的海风的味道,令人回味.大概一个小时,环岛骑行算是结束了.
接下来是坐快艇去东升岛游览.快艇这辈子是第一次坐,在浪里前行,有一种时刻在漂移的感觉,第一次感觉浪得感觉很好,哈哈.不过坐船边的哥们就惨了,裤子全湿了.登岛之后,游览了岛上的民居,然后临近回去的时候,大雨如期而至,还好旁边有一座宾馆可以用来避雨.好不容易等到雨小了,总算是再次乘坐快艇回到了大陆的怀抱.
最后就是野炊环节,我的任务很简单,生火,杀鸡.作为一个常年做饭的老手我自然是很快搞定.我们分了几组团队,我们这组领导比较多,那有些人就干得格外卖力,有一个妹子一看就是没咋做过饭的,硬是杵在那里做了几个菜,十几分钟做的几个菜,硬是做了半个多小时,我心里其实明白她为啥要这么做.本来大家是叫我做,我看她在哪里做的格外卖力,也没有和她争,谁不想吃现成的呢,嘿嘿.
饭吃完了,雨也停了,也该回去了.总的来说,这次活动没有想象中的那么令人满意,不够纯粹.我庆幸自己是一个做技术的程序员,不用为了一些其他东西额外去伤脑筋.做人还是纯粹点,会更洒脱快乐-.

4.20心情总结

今天下班之后大概七点左右的样子,一个团队的同事发微信跟我说不想干了,原因是干的活多,拿的太少.
其实细想一下,他作为一个刚毕业不到一年的大学生,确实是做了不少工作,比较复杂的任务他都完成了.对比另外几个搞后台的同级毕业生,他的确出类拔萃.
最直接的导火索是,马上七月份面临毕业生定岗,工资会有所调整,但是老大透露的意思最多只能涨一部分.这完全不能达到他希望的要求.因为对比其他大学生,他干的活,做的事是多且不错的.所以他就有些心灰意冷的感觉.
其实我也有相同的感觉,单纯作为一个从事技术的人来说,工资的涨幅不按能力而按年限,本来就不合理.这里做了几年的老员工,工资每年涨固定的数额,而且数额很小.当听到这个事的时候,我心里也发凉.我这才真正的意识到,这个公司可能不太重视技术这块,对待技术有一种莫名的偏颇.
这是我来深圳的第一家公司,我很感激这家公司给我的良好环境,让我有时间沉淀自己的技术.但是,某些随着时间推移,我也慢慢感受到其中的政治意味是何等强烈.最近听到了一个"站队"的概念,我挺想笑,技术是纯粹的,站到哪一队难道就会有影响,该干活你还是得干啊.作为一个追求技术发展的人来说,我不认为这是一个好的科技公司,好的科技公司是不会频繁请外包的,核心都是在自己手里,不容他人染指.
最后总结:最迟明年,可能我也会走.找一个在技术上有发展的地方,证明自己!
