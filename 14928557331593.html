4.22重拾java之开端

2017/4/22

前一段时间的纠结可以告一段落了,我还是选择了重拾java这一条路,毕竟这曾经是让我入门编程的语言.
下了一篇电子书,快速得浏览了一下一些基础和面向对象的实例之后,之前有些疑惑也豁然开朗.同时心中也难免产生一些和OC的对比.举个栗子:

OC写一个构造方法
- (instance)initWithName:(NSString *)name{
    self = [super init];
    if (self){
        self.name = name;
    }
    return self;
}

Java写一个构造方法:
class TestDemo{
    public TestDemo (String name){
        this.name = name;
    }
}
-----------分隔线---------------
OC中写一个单例:
+(DBManager *)singleton; (.h文件中)
.m文件中的实现:
+(DBManager *)singleton{
　　Static DBManager *singleton = nil;
      static dispatch_once_t token;
      dispatch_once(&token,^{
              if(singleton == nil){
　　         singleton = [[DBManager alloc]init];
        }
   } );
  return singleton;
}

Java中写一个单例:
class Singleton{
    private static Singleton instance = new Singleton();
    private Singleton(){   
    }
    public static Singleton getInstance(){
        return instance;
    }
}

其实都是异曲同工.

->关于self和super都必须放在放在构造方法的第一行.

->final关键字修饰的类,方法,属性不能被继承,重写和修改,修饰的常量必须是'大写'.

->抽象类中,需要在类名前加一个abstract,声明抽象方法只需声明无须事先,必须在权限后加abstract关键字.并且抽象类不能用final关键字,必须被继承,抽象方法不能用private关键字,因为必须被子类重写.

->interface中定义的是全局常量和public抽象方法(不写public也默认为此,而非default)

->instanceof类似OC中的isKindOfClass,可判断实例类型

->模板设计:父类定义一些可能用到的方法作为模板,子类按照模板实现方法就行了.

->避免空指针小技巧,将str.equalsto("123")改成"123".equalsto(str)

->接口也可以被实例化,接受的是实现接口抽象方法的类,类似于父类和子类的关系

->java中的适配器模式:接口方法过多,那么声明一个抽象类实现接口所有接口方法,但是该类实现的方法都为空实现,最后声明一个类继承这个抽象类,实现需要的方法.
